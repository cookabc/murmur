# Voice Input for macOS 技术评估与优化方案

## 1. 评估结论

这个项目已经完成了一个可演示的产品原型：录音、ASR、结果展示、历史记录和跨应用插入这些关键能力已经串起来了。但当前实现仍然偏“原型工程”，距离稳定、可维护、可长期迭代的本地语音输入工具还有明显差距。

当前最需要解决的问题不是新功能不足，而是主链路存在高风险实现：录音进程生命周期管理、热键机制空实现、ASR 阻塞调用、系统集成错误处理不足。这些问题会直接影响真实用户使用体验与后续维护成本。

## 2. 当前实现的优点

### 2.1 方向正确

- 选题明确，产品核心价值清晰
- 本地处理链路符合隐私诉求
- 以跨应用输入为目标，有明显使用场景

### 2.2 模块边界初步成立

后端已经按能力拆分为：

- `audio.rs`
- `asr.rs`
- `clipboard.rs`
- `hotkey.rs`
- `settings.rs`
- `lib.rs`

这为后续重构提供了起点，说明代码还没有发展到不可拆分的程度。

### 2.3 前后端交互简单直接

- 前端使用少量状态完成主交互
- Tauri `invoke` 命令较清晰
- 产品操作路径短，适合桌面小工具形态

## 3. 主要问题与风险

### 3.1 P0 级问题：会影响主链路稳定性

#### A. 录音进程句柄被主动丢弃

位置：`src-tauri/src/audio.rs`

问题：

- 启动 ffmpeg 后使用 `std::mem::forget(child)` 丢弃进程句柄
- 应用失去了对当前录音进程的精确控制能力

影响：

- 无法只停止当前录音进程
- 多次录音可能导致进程管理混乱
- 难以实现可靠的停止、超时和异常清理

#### B. 停止录音依赖 `pkill ffmpeg`

位置：`src-tauri/src/audio.rs`

问题：

- 停止逻辑通过全局 `pkill -INT ffmpeg` 完成
- 这会影响系统中所有 ffmpeg 进程，而不仅是当前应用启动的实例

影响：

- 用户其他依赖 ffmpeg 的任务可能被误杀
- 工具行为不可预测
- 这是一类典型的系统级副作用 bug

#### C. ASR 调用是同步阻塞式外部命令

位置：`src-tauri/src/asr.rs`

问题：

- `Command::new("coli").output()` 为阻塞执行
- 没有超时、取消、中断和并发策略

影响：

- 转写耗时长时可能卡住界面或命令调用线程
- 外部工具异常时恢复路径弱

### 3.2 P1 级问题：产品承诺和实现不一致

#### A. 热键注册基本未完成

位置：`src-tauri/src/hotkey.rs`

问题：

- `register_hotkey` 仅保存字符串，不做真正注册
- UI 中“Change Hotkey”按钮也明确提示未完整实现

影响：

- README 中的核心卖点未被真实兑现
- 用户会把热键能力当作稳定功能，但实际并非如此

#### B. 输入法式定位与普通窗口实现不一致

位置：`src-tauri/tauri.conf.json`

问题：

- 已配置 tray icon，但没有完整托盘菜单与后台工作流
- 当前交互核心还是可见主窗口，主流程仍依赖面板式 UI

影响：

- 产品定位模糊
- 用户对“menu bar app”的预期会落空

#### C. AI Polish 是暴露给用户的占位能力

位置：`src-tauri/src/asr.rs`

问题：

- 设置项和前端 UI 暴露了该能力
- 但后端注释说明 `coli asr` 不支持该参数

影响：

- 产品文案与实际能力不一致
- 用户可能认为开关无效或产品不可信

### 3.3 P2 级问题：可维护性和演进风险

#### A. 状态管理方式不统一

位置：`src-tauri/src/lib.rs`

问题：

- 有的状态是 `Arc<Mutex<T>>`
- 有的状态是 `Mutex<T>`
- `AudioRecorder` 自己又包了一层内部 `Arc<Mutex<_>>`

影响：

- 心智负担偏高
- 后续维护时容易引入锁使用不一致问题

#### B. 存储层是直接读写 JSON 文件

位置：`src-tauri/src/settings.rs`

问题：

- 每次都完整读取和回写 JSON
- 没有损坏恢复、原子写入、版本迁移和并发保护方案

影响：

- 配置文件损坏后恢复成本高
- 并发命令下存在脏写风险

#### C. 剪贴板与 AppleScript 路径耦合较深

位置：`src-tauri/src/clipboard.rs`

问题：

- 系统脚本策略直接写死在模块内
- 特殊字符转义和权限失败路径处理较弱

影响：

- 兼容性问题难排查
- 用户看到的错误信息不够具体

#### D. 前端源码结构不可维护

位置：`dist/app.js`

问题：

- 当前仓库以构建产物作为主要前端源码
- `src/` 目录为空

影响：

- 迭代前端逻辑会持续修改产物文件
- 构建来源不清晰，难以团队协作

## 4. 设计层面的优劣总结

### 4.1 优势

- 产品目标简单，核心路径短
- 技术栈轻量，Tauri 适合此类工具
- 系统集成思路符合 macOS 小工具特征

### 4.2 劣势

- 外部进程和系统能力调用过于粗糙
- 产品承诺先于实现，导致可信度下降
- 工程结构还没有形成稳定的分层边界
- 测试与自检几乎空白

## 5. 优化与重构方案

### 5.1 第一阶段：先修主链路稳定性

目标：让应用在“录音 -> 停止 -> 转写 -> 粘贴”这条路径上足够可控。

#### 建议 1：重写录音进程管理

改进方案：

- `AudioRecorder` 持有当前录音进程句柄
- 明确状态：`Idle | Recording { child, path, started_at } | Stopping`
- 停止时只终止当前子进程
- 增加录音超时与清理逻辑

验收标准：

- 连续录音 10 次无残留 ffmpeg 进程
- 停止录音不会影响系统其他 ffmpeg 任务

#### 建议 2：将 ASR 调用改为异步、可超时

改进方案：

- 使用 `tokio::process::Command`
- 增加超时控制
- 支持取消当前识别任务
- 明确 stdout / stderr 解析规则

验收标准：

- 外部命令失败时 UI 可恢复
- 长音频或异常进程不会永久卡死应用

#### 建议 3：补齐启动依赖自检

改进方案：

- 启动时检查 `ffmpeg`、`coli`、必要权限
- 给出用户可执行的修复提示
- 在 UI 中明确标识当前依赖状态

验收标准：

- 缺依赖时用户能快速知道原因和解决方式

### 5.2 第二阶段：兑现产品承诺

目标：把 README 中最核心的卖点真正落地。

#### 建议 4：实现真实热键注册系统

改进方案：

- 基于 `tauri-plugin-global-shortcut` 完成热键注册、注销、切换
- 增加冲突检测和失败提示
- 设置保存后立即应用新热键

验收标准：

- 修改热键后无需重启即可生效
- 冲突或非法组合可明确提示

#### 建议 5：明确输入法式产品模型

有两种方向，必须选一条：

##### 方向 A：输入法式桌面工具

- 全局热键承担主要入口
- 主窗口默认隐藏，只在需要时显示
- 托盘承担状态入口、最近结果查看和退出

##### 方向 B：保留控制面板工具定位

- 删除“像输入法一样工作”的表述
- 承认当前是有主窗口的小工具应用

既然产品目标已明确为本地语音输入法式工具，就应选择 A。

#### 建议 6：处理 AI Polish 的产品诚实性问题

改进方案：

- 若暂时无实现，先从 UI 移除或标记为实验功能
- 若要保留，则在转写后增加本地文本后处理链路

验收标准：

- 用户看到的功能开关必须对应真实效果

### 5.3 第三阶段：结构化重构

目标：建立可维护的工程分层。

#### 推荐分层

##### A. Command 层

只负责 Tauri 命令输入输出，不承载业务逻辑。

例如：

- `commands/audio_commands.rs`
- `commands/asr_commands.rs`
- `commands/settings_commands.rs`

##### B. Service 层

负责具体业务能力。

- `services/recorder_service.rs`
- `services/asr_service.rs`
- `services/paste_service.rs`
- `services/hotkey_service.rs`
- `services/history_service.rs`

##### C. Infrastructure 层

负责系统命令、文件存储、权限探测等细节。

- `infra/process_runner.rs`
- `infra/store.rs`
- `infra/clipboard.rs`
- `infra/macos_automation.rs`

##### D. Domain 层

抽离核心模型和状态。

- `RecordingSession`
- `TranscriptionResult`
- `AppSettings`
- `HistoryEntry`
- `AppHealth`

#### 预期收益

- 业务逻辑不再散落在 Tauri 命令和系统脚本之间
- 测试可以针对 service 层完成
- 后续替换底层实现成本更低

### 5.4 第四阶段：前端可维护化

目标：让 UI 成为可持续演进的代码，而不是直接编辑构建产物。

建议：

- 把 `dist/` 视为构建输出，而不是源码目录
- 恢复或重建 `src/` 前端源码
- 抽离状态管理、API 层、视图层
- 为依赖检查、录音中、转写中、失败态设计更明确的反馈

如果不想引入复杂框架，保留原生 JS 也可以，但至少应恢复源码结构。

## 6. 测试与验证方案

### 6.1 单元测试

- 设置读写
- 历史记录增删改查
- ASR 响应解析
- 粘贴文本转义

### 6.2 集成测试

- 录音开始 / 停止流程
- `coli` 不可用时的失败路径
- 自动粘贴成功 / 失败回退
- 热键注册与热键切换

### 6.3 手工回归清单

- 连续录音 10 次
- 长文本粘贴
- 含换行、引号、中文的文本粘贴
- 麦克风权限拒绝后恢复
- 无 `ffmpeg` / 无 `coli` 场景
- 全局唤起、窗口按需显示、退出流程

## 7. 推荐执行顺序

### P0

- 修录音进程管理
- 修停止录音策略
- 修 ASR 阻塞和超时
- 加启动自检

### P1

- 实现热键注册与热键修改
- 明确全局唤起 / 托盘 / 主窗口产品模型
- 处理 AI Polish 真伪不一致问题

### P2

- 重构后端分层
- 重构前端源码结构
- 引入更完整的错误模型和日志

### P3

- 国际化
- 历史搜索
- 输入源选择
- 启动自动运行

## 8. 最终建议

这个项目不应该继续沿着“边加功能边修补”方式推进。更合理的路径是：

1. 先把主链路修稳。
2. 再把热键和输入法式交互这两个产品承诺真正兑现。
3. 然后做一次中等规模重构，把它从原型工程升级成可维护项目。

如果只做功能叠加而不处理录音进程、热键空实现和阻塞调用，后续每增加一个功能，都会进一步放大现有结构的问题。
